# Linux多线程服务端C++编程

## 1 线程安全的对象生命期管理

### 1.2 对象的创建很简单
构造函数中不要泄露this指针，因为在构造期间对象还没完成初始化，如果this泄露给了其他对象，那别的线程就有可能访问到这个半成品对象  
可以使用二段式构造，构造结束后把这个this注册到需要的对象上  

### 1.3.1 mutex 不是办法
A线程在析构class时，B线程调用了即将被析构的成员函数导致的race  

### 1.3.2 作为数据成员的 mutex 不能保护析构
作为数据成员的mutex无法保护析构  
同时读写一个class的两个对象，有潜在死锁的可能，可以比较多个mutex的地址，始终先锁地址小的mutex[p8]  

### 1.4 线程安全的 Observer 有多难
该小节主要讲述了析构时机导致实现线程安全的类是比较困难的  
通过sleep制造时间窗口，让主线程执行到`x->update()`时，x正在析构[p10]

### 1.5 原始指针有何不妥
内建指针类型无法知道是否存活（指针置位NULL也不行，书上边讨论过）的原因，如果有一个对象能知道目标对象是否存活就好了  
基于此引入一层间接性来解决这个问题：shared_ptr/weak_ptr  

### 1.8 应用到 Observer 上
@Observer_safe.cc解决了[p16]的竞态条件  
如果把`std::vector<boost::weak_ptr<Observer> > observers_;`改成`std::vector<boost::shared_ptr<Observer> > observers_;`会怎么样？  
如果改，`s->register_(shared_from_this());`直接push shared_ptr，Foo对象被强引用，计数为2（循环引用）。退出作用域销毁时计数变为1，无法被销毁  
不改的话weak_ptr对Foo对象不是强引用，不增加计数，只有对象还活着才会提升为shared_ptr；而提升为share_ptr后这个强引用是栈上对象，其他线程无法销毁；退出作用域销毁时计数变为0，正常被销毁 
`boost::shared_ptr<Observer> obj(it->lock());`这一步提升是线程安全的   

列出了一些问题：  
1. 侵入性
2. 不完全线程安全
3. 锁争用
4. 死锁

### 1.9 再论 shared_ptr 的线程安全
shared_ptr本身的线程安全需要锁保证  
写全局shared_ptr时为了避免在临界区内析构，创建一个栈上变量swap，这样临界区内原对象不会析构，退出临界区结束调用后自动析构

2. 可以当函数参数传递引用，不必拷贝

### 1.11 对象池
@Factory.cc  

version1情况下stocks_是shared_ptr，永远不会被销毁；应该改为version2的weak_ptr

```c++
pStock.reset(new Stock(key),
  boost::bind(&StockFactory::deleteStock, this, _1));
```
让它在析构Stock *p时调用本 StockFactory 对象的 deleteStock 成员函数  
这里传入了StockFactory的this指针，如果StockFactory提前被析构，还是未定义行为  

那么问题应该就是如何保证StockFactory这个对象的生命周期，应该要祭出惯用的shared_ptr大法，而enable_shared_from_this可以让this指针变成shared_ptr  

```c++
pStock.reset(new Stock(key),
  boost::bind(&StockFactory::deleteStock,
  shared_from_this(),
  _1));
```
通过上述方法，boost::function里就有一份StockFactory的shared_ptr拷贝了，不需要担心StockFactory被析构的危险了；但是似乎StockFactory的生命周期被延长了  

```c++
pStock.reset(new Stock(key),
  boost::bind(&StockFactory::weakDeleteCallback,
    boost::weak_ptr<StockFactory>(shared_from_this()),
    _1));
```
通过将shared_from_this()强转为weak_ptr，可以实现弱回调技术

## 3 多线程服务器的适用场合与常用编程模型

### 3.5.2 单线程程序的优缺点

如果要在一台多核机器上提供一种服务或执行一个任务，可用的模式有:  
1. 运行一个单线程的进程
2. 运行一个多线程的进程
3. 运行多个单线程的进程
4. 运行多个多线程的进程

模式1是不可伸缩的，不能发挥多核机器的计算能力  
模式3是目前公认的主流模式。它有以下两种子模式:
3a 简单地把模式1中的进程运行多份攻  
3b 主进程 + woker 进程，如果必须绑定到一个 TCP port，比如 httpd+fastcgi
模式2是被很多人所鄙视的，认为多线程程序难写，而且与模式 3 相比并没有什么优势  
模式4更是千夫所指，它不但没有结合2和3的优点，反而汇聚了二者的缺点  

Event loop有一个明显的缺点，它是非抢占的 ( non-preemptive )。假设事件a的优先级高于事件b，处理事件 a需要1ms，处理事件b需要 10ms。如果事件b稍早于a发生，那么当事件a到来时，程序已经离开了poll(2)调用，并开始处理事件b。事件 a要等上 10ms才有机会被处理，总的响应时间为11ms。这等于发生了优先级反转。这个缺点可以用多线程来克服，这也是多线程的主要优势。
